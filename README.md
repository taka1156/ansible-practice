# ansible-practice

A practice environment for learning and experimenting with Ansible using **VS Code Dev Containers**.
No virtual machines are required — Docker containers alone provide an “Ansible execution environment (bastion)” and a “managed target server (target),” so you can start writing and running playbooks right away.

## Features

- Just `Reopen in Container` with the VS Code Dev Containers extension to set up the whole environment
- Two-container setup: an Ansible control node (bastion) and an SSH-managed node (target)
- SSH key pairs are automatically generated and distributed on startup (no manual key creation/copying needed)
- Includes tooling such as `ansible-lint` and `secretlint`
- Pre-configured Japanese locale (`ja_JP.UTF-8`) and timezone (`Asia/Tokyo`)

## Prerequisites

- Docker / Docker Compose
- Visual Studio Code
- VS Code extension: [Dev Containers](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-containers)

## Setup

1. Clone the repository
   
   ```bash
   git clone https://github.com/taka1156/ansible-practice.git
   cd ansible-practice
   ```
1. Open it in VS Code and run `Dev Containers: Reopen in Container` from the command palette
- Two containers, `ansible-bastion` (Ansible execution environment) and `ansible-target` (managed server), will start
- `ansible-target` generates an SSH key pair, which is automatically copied to `ansible-bastion`
1. Once the containers are up, work from a terminal inside `ansible-bastion`

## Usage

### Connectivity check

```bash
make ping
```

Runs `ansible -i ansible/inventory.ini ubuntu_servers -m ping` to verify the connection to `ansible-target`.

### SSH into the target server

```bash
make ssh
```

### Running a playbook

```bash
ansible-playbook -i ansible/inventory.ini your_playbook.yml
```

### Lint

```bash
ansible-lint
```

## Directory structure

```
.
├── .devcontainer/
│   ├── devcontainer.json        # Dev Container configuration
│   ├── docker-compose.yml       # Defines the bastion / target containers
│   ├── ansible_bastion/
│   │   └── Dockerfile           # Ansible execution environment
│   ├── ansible_target/
│   │   ├── Dockerfile           # SSH-managed server
│   │   └── entrypoint.sh        # Generates SSH key & registers authorized_keys
│   └── scripts/
│       └── sync-ssh-key.sh      # Syncs the key generated by target to the bastion
├── ansible/
│   └── inventory.ini            # Inventory (ubuntu_servers group)
├── ansible.cfg                  # Ansible config (remote_user, private key path, etc.)
├── Makefile                     # ssh / ping shortcuts
└── README.md
```

## Architecture overview

|Container        |Role                         |Contents                                                 |
|-----------------|-----------------------------|---------------------------------------------------------|
|`ansible-bastion`|Ansible execution environment|git, ansible, ansible-lint, secretlint, cloudflared, etc.|
|`ansible-target` |Managed server               |openssh-server, python3, `ansible` user (NOPASSWD sudo)  |

Both containers are connected via the Docker network `lab-net`, allowing `ansible-bastion` to SSH into `ansible-target` using the hostname `ansible-target`.
